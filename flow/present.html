Three Paths to a "Company Profile" Bot

A comparative look at AWS Bedrock, Google Dialogflow, and n8n for building intelligent, data-driven conversational agents.

June 16, 2025

The Core Conversation

Our goal is to build a chatbot that can handle this multi-turn dialogue.

U

"Tell me about Apple Inc."

Apple Inc. designs, manufactures, and markets smartphones, PCs, tablets, and wearables...

Would you like to know how it makes money or who its key executives are?

U

"How does it make money?"

Apple's primary revenue source is the iPhone. Other significant streams include its Services division (App Store, iCloud), Mac, iPad, and Wearables.

Approach 1

AWS Bedrock Agent

An LLM-native approach where the agent orchestrates "tools" (Lambda functions) to perform actions based on natural language understanding.

Architecture

User → Bedrock Agent → Lambda Tool → External API

Key Components

  • Bedrock Agent: The "brain" that manages conversation flow.
  • Lambda Function: A "tool" containing business logic to fetch data.
  • Agent Instructions: A natural language prompt telling the agent how to behave and which tools to use.

Best For

  • Teams already invested in the AWS ecosystem.
  • Rapid prototyping with complex, multi-step reasoning.
  • Developers comfortable with serverless architecture.

AWS Bedrock: Implementation

Implementation revolves around defining a Lambda tool and instructing the agent.

1. Lambda Tool Input Schema

{
  "ticker": "string",
  "queryType": "string" // e.g., "businessSummary"
}

2. Lambda Tool Output Schema

{
  "ticker": "AAPL",
  "revenueStreams": [
    { "segment": "iPhone", "percentage": 52.0 },
    { "segment": "Services", "percentage": 20.0 }
  ]
}

3. Agent Instructions (Natural Language)

// In the Bedrock Agent Console... When a user asks how a company makes money, use the CompanyProfileTool with the queryType 'revenueStreams'. Always identify the company's stock ticker.

Approach 2

Google Dialogflow CX

A structured, state machine-based platform where conversation flows are explicitly designed using pages, intents, and routes.

Architecture

User → Dialogflow CX → Webhook → External API

Key Components

  • Intents: What the user wants to do (e.g., `get_company_revenue`).
  • Entities: Keywords to extract (e.g., `@company-name`).
  • Flows & Pages: Visual representation of the conversation path.
  • Webhook: Backend code (e.g., Cloud Function) to fetch dynamic data.

Best For

  • Structured, goal-oriented conversations (e.g., customer support, booking).
  • Visual designers and teams who prefer low-code environments.
  • Projects requiring explicit control over every conversational turn.

Dialogflow CX: Implementation

Implementation focuses on defining intents and a webhook for data fetching.

1. Define an Intent

Intent Display Name: get_company_revenue

Training Phrases:

  • "How does @company-name make money?"
  • "What are @company-name's revenue streams?"

2. Design a Page Route

IF intent IS get_company_revenue

THEN

CALL webhook companyInfoWebhook

RESPOND with "According to their filings, $session.params.revenue_summary"

3. Webhook (Google Cloud Function) Logic

// Pseudocode
async function companyInfoWebhook(req, res) {
  // 1. Get company name from Dialogflow request
  const companyName = req.body.sessionInfo.parameters['company-name'];
  
  // 2. Call external API to get data
  const apiData = await fetchData(companyName);

  // 3. Format response for Dialogflow
  const response = {
    session_info: {
      parameters: { revenue_summary: apiData.summary }
    }
  };
  
  res.send(response);
}
Approach 3

n8n Workflow Automation

A flexible, node-based workflow tool where you chain services together. The "chatbot" is a workflow triggered by a webhook.

Architecture

Webhook → LLM Node → HTTP Node → LLM Node

Key Components

  • Webhook Node: The entry point for user messages.
  • LLM Node: Used for NLU (intent extraction) and NLG (response generation).
  • HTTP Request Node: Calls any external API to get data.
  • Code/Flow Logic: The visual connection between nodes defines the process.

Best For

  • Integrating many different APIs and services.
  • Engineers who think in terms of data flows and ETL.
  • Maximum flexibility and control over the entire process.

n8n: Implementation

Implementation is a chain of nodes, each performing a specific task.

Node 1: LLM (Extract Intent)

Takes raw user query and outputs structured JSON.

PROMPT: Analyze the text `{{ $json.body.query }}` and return JSON with 'ticker' and 'queryType'.
↓

Node 2: HTTP Request

Uses JSON from the previous node to call the financial API.

URL: `https://financial-api.com?ticker={{ $node["LLM (Extract Intent)"].json.ticker }}`
↓

Node 3: LLM (Generate Response)

Takes API data and the original query to form a natural language answer.

PROMPT: Based on this data `{{ $node["HTTP Request"].json }}`, answer this question: `{{ $node["Webhook"].json.body.query }}`

Comparison of Approaches

Factor AWS Bedrock Agent Google Dialogflow CX n8n
Core Paradigm LLM-native Orchestration Visual State Machine Workflow Automation
Learning Curve Moderate (Requires AWS knowledge) Low to Moderate (Visual) Low (Node-based logic)
Flexibility High (Code-based tools) Moderate (Structured flows) Very High (Integrates anything)
Control Abstracted (Agent decides) Explicit (You define all paths) Granular (You control every step)
Best For... Complex reasoning & AWS shops Predictable, guided tasks Complex data integration

Conclusion & Next Steps

Each platform offers a powerful but different path to building a conversational agent. The best choice depends on your team's existing skills, project requirements, and desired level of control.

Recommendation

For the InHouse AI Chatbot project, AWS Bedrock Agent aligns best with the existing PRD, leveraging the AWS-native ecosystem and providing a scalable foundation for future tool additions.
